Observations UX

Basées sur index.html et css/style.css. Rien de cassé — des frictions à lisser.

Onboarding

Double wizard, même intention

Il existe deux parcours parallèles : #first-resource-onboarding (avec .fro-*) et le #signup-overlay (avec .su-*), qui posent tous les deux « Quel type de ressource ? → Nom famille → Nom ressource ».

  • Fusionner en un seul flux stateful, piloté par onboarding.service.js.
  • Réutiliser les mêmes composants (photo picker, champ nom, progress dots).
  • Économie estimée : ~400 lignes de HTML/CSS/JS.
Planning / Réservation

Le wizard masque trop le calendrier

Le toggle body.planning-booking-focus cache header, onglets ressources, et agrandit la sticky‑stack. Bien pour la concentration, mais l'utilisateur perd le contexte « quelle ressource ? ».

  • Garder un bandeau compact ressource + dates choisies en haut.
  • Le CTA « × fermer » devrait offrir « Garder mes dates / Tout effacer ».
  • Ajouter un fil d'Ariane Dates → Personnes → Qui? → Destination.
Dashboard

Densité d'information inégale

La carte hero affiche « Cette semaine » + strip 7 jours + CTA, c'est bien. Mais juste en dessous, les infos maison / voiture se dédoublent (#house-info-card + #car-info-card + #car-info-grid) avec beaucoup de « — » quand les champs ne sont pas remplis.

  • Masquer les lignes vides plutôt que d'afficher « — ».
  • Ajouter un progress de complétion du profil ressource (déjà amorcé par famresa-completion.js).
  • Regrouper Assurance + CT + Réservoir dans une carte unique « État du véhicule ».
Gamification

XP présent dans le DOM mais caché

Le hero XP (#xp-hero-card) est maintenu en display:none « pour compat JS ». Soit on l'assume et on le réintroduit (bandeau compact sur dashboard), soit on retire complètement le code mort.

  • Décision produit à prendre. Tel quel = complexité sans valeur.

Deux exemples concrets

Pour illustrer l'impact des propositions 01 et 04 — sans toucher au code du projet.

Avant — firestore.rules (extrait)
match /reservations/{docId} {
  allow read: if signedIn();
  allow create: if signedIn()
    && request.resource.data.keys()
        .hasAll(['profil_id','startDate','endDate']);
  …
}
Tout compte authentifié peut lire toutes les réservations de toutes les familles.
Après — scope par famille
function isMyFamily(fid) {
  return exists(/databases/$(database)/documents/
    famille_membres/$(request.auth.uid + '_' + fid));
}

match /reservations/{docId} {
  allow read: if signedIn()
    && isMyFamily(resource.data.famille_id);
  allow create: if signedIn()
    && isMyFamily(request.resource.data.famille_id)
    && request.resource.data.profil_id == request.auth.uid;
  …
}
Suppose un doc famille_membres/{uid}_{familleId} indexable. Nécessite une petite migration + un seed côté signUp.
Avant — js/booking.js
const ref = await reservationsRef().add({
  profil_id: currentUser.id,
  ressource_id: selectedResource,
  startDate, endDate, …
});
// puis : appel UI de célébration
Pas de vérification d'overlap atomique. Deux clics simultanés = deux résas qui se chevauchent.
Après — appel service
const res = await reservationService.create({
  resourceId: selectedResource,
  range: { start, end, startHour, endHour },
  booker, participants
});
if (res.conflict) showConflictSheet(res.conflict);
else celebrate(res.reservation);
Le service fait runTransaction() : lit les résas de la plage, valide capacité, écrit si libre. UI reste bête.

Proposition de roadmap

3 itérations de 2 semaines chacune. Aucune casse utilisateur — tout se fait derrière les services existants.

Sprint 01 · Sécurité

Durcir les règles

  1. Ajouter famille_id sur toute réservation/ressource et scoper les lectures.
  2. Activer App Check (reCAPTCHA v3 web).
  3. Suite de tests rules-unit-testing (10 cas).
  4. Auditer quotas Firestore + budget alerts.
Sprint 02 · Architecture

Finir la migration services

  1. Extraire tout db.collection() de js/ vers /src/modules/*.
  2. reservation.service avec runTransaction pour overlap.
  3. Supprimer les branches legacy (users, families/{id}/bookings).
  4. Découper style.css et booking.js.
Sprint 03 · Polish

UX + PWA + DS

  1. Fusion des deux onboardings en un seul.
  2. Fil d'Ariane dans le wizard de réservation.
  3. Page /styleguide.html (tokens, composants).
  4. Audit Lighthouse & correctifs a11y.

Quick‑wins (moins d'½ journée chacun)

DX

Un CLAUDE.md / .codex.md à la racine

Vos docs AI_GUIDE.md / DEV_RULES.md sont parfaites — mais un CLAUDE.md à la racine force toute IA à les lire en premier. Ajouter aussi npm run lint (ESLint + stylelint) et npm run check.

Qualité

Ajouter un CHANGELOG.md

version.js existe déjà et alimente le cache SW. Documenter chaque bump évitera les incidents côté PWA (runbook déjà bien écrit — le relier au changelog).

Sécu

Retirer admin-profiles-audit.html du dépôt public

Même en lecture seule + login, une page d'audit servie depuis GitHub Pages n'a rien à faire au même endroit que le reste. Déplacer sous /admin/ protégé par un noindex + Firebase Hosting rewrites.

Perf

Pré‑load des polices + font-display: swap

Aujourd'hui media="print" onload="this.media='all'". Préférer <link rel="preload" as="font" crossorigin> pour les 2 graisses critiques (400 / 600) + fallback local -apple-system déjà en place.

✓

TL;DR — FamResa est un projet sain, avec une ambition d'architecture claire et une UX mobile soignée. Les trois chantiers à lancer en priorité : (1) scoper les règles Firestore par famille, (2) finir la migration js/ → src/modules/ avec une vraie transaction sur les réservations, (3) découper style.css et booking.js. Le reste est du confort — important mais pas bloquant.
